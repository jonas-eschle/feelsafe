# Design Rationale — Guardian Angela

Design rationale for Guardian Angela. Documents the reasoning behind key architectural and UX decisions.

---

## App Identity

### Name & Branding
- **Final name**: Guardian Angela (wordplay on "guardian angel" + "Ask for Angela" safety campaign)
- **Tagline**: "Your angel's got your back."
- **Application ID**: `com.guardianangela.app`
- **Pride branding**: Permanent, mandatory, non-negotiable. No non-pride variant.
- **Logo**: Pride-flag gradient angel with feathered wings, shield body (cutout gap between wings and shield), and halo. Generated by `tool/generate_icon.dart`.
- **Trademark risk**: Ask for Angela CIC holds "Ask for Angela" trademark; "Guardian Angel" has prior apps in namespace. Recommendation: seek partnership before commercial launch.

## Platform & Compatibility

### Target Platforms
- **Platforms**: Android + iOS only. No web, no desktop.
- **Rationale**: 95%+ device coverage on both platforms; web/desktop out of scope.

### Minimum Versions
- **Android**: Minimum SDK 26 (API 26, Android 8.0), target SDK 35
  - Rationale: API 26 required for notification channels (critical feature)
  - Target 35 for Play Store compliance
- **iOS**: Minimum 16.0
  - Rationale: Modern notification and background execution APIs; older versions lack required capabilities
- **Rationale overall**: Balances modern feature support with reasonable device coverage

### Platform Limitations (Documented)
- **iOS SMS limitation**: Opening Messages app requires manual Send button press (cannot auto-send)
- **iOS call limitation**: Always shows call confirmation dialog; cannot bypass to auto-dial
- **iOS hardware button**: Not supported (greyed out in settings)
  - Reason: Volume buttons only work in foreground; headphone remote requires connected headphones
- **iOS app kill detection**: Only detectable at next launch, not in background
- **whatsapp/Telegram deep links**: Cannot initiate voice calls (removed as phone call channels)

## Engine Architecture

### State Machine Design
- **Core component**: Pure Dart state machine with zero Flutter dependencies (`SessionEngine`)
- **Advantages**: Testable, simulatable, platform-agnostic, deterministic
- **Session lifecycle**:
  - Starts on first user interaction (not on "Start" button)
  - Only one session active at a time
  - Emits `ChainEventData` via `Stream`
  - Manages timer-driven escalation steps

### Timing Model
- **3-phase timing per step**: wait → duration → grace
  - **Wait**: Initial delay before starting the step
  - **Duration**: How long the step is active (e.g., call rings, alarm sounds, countdown displays)
  - **Grace**: Additional time after duration expires to perform optional action (e.g., "I'm Safe" disarm)
- **Jitter**: ±20% random jitter applied to timers
- **Speed multiplier**: Configurable for simulation/testing (up to 1000x)
- **Randomization**: Per-field in engine via config keys (`randomizeInterval`, `randomizeRingDuration`, etc.)

### Check-In Methods
1. **Walk Mode**:
   - User holds a button to keep session active
   - Releasing the button starts the grace period
   - User must re-press before grace expires to reset

2. **Date Mode**:
   - Periodic disguised reminders via notifications
   - User must interact within grace period
   - `waitSeconds`: interval between confirmed check-ins AND initial delay

### Disguised Reminder Timing
- **waitSeconds**: Time interval between confirmed check-ins AND initial delay before first reminder
- **retryCount**: Number of retries (N retries = N+1 total attempts)
  - Example: retryCount=2 means 3 total reminders (initial + 2 retries)
- **Rationale**: Single `waitSeconds` simplifies configuration versus separate retry intervals

## Distress Modes (Pivot 3 — distress is a Mode)

### Distress is a Mode
- **No separate model.** A distress chain is the `chainSteps` of a regular `SessionMode`
  flagged with `isDistressMode = true`. Distress modes live alongside regular modes in
  `modes.json`.
- **Mode reference**: Each `SessionMode` has `distressModeId` (String?). null =
  inherit `AppDefaults.defaultDistressModeId`. If neither resolves, mode validation blocks
  session start.
- **Single distress chain per mode**: All three distress triggers (hardware panic, duress
  PIN, wrong PIN threshold) fire the same resolved distress mode's chain. No per-trigger
  customization.

### Design Rationale
- **Why a Mode?** Distress chains needed exactly the editing affordances modes already had (timing, retry, jitter, full per-step config). Adding `isDistressMode` was cheaper than maintaining a parallel `DistressChain` aggregate, repository, and editor screen.
- **Why one chain per mode?**: Simplifies the trigger system — one consistent response regardless of what triggered distress.

## Disarm Mechanism

### Gesture & Interaction
- **Gesture**: Swipe left-to-right slider (not button press)
- **Rationale**: Swipe prevents accidental taps; gesture-based is harder to trigger by touch alone
- **Stealth text**: "No Angela needed" (fully configurable)
- **Behavior on disarm**: Always resets to step 0 and clears miss count
- **Availability**: Only during grace period (context-dependent)

## Hold Button

### Styles
- **3 styles available**:
  1. `largeButton`: Large, prominent button (good for one-handed use, default)
  2. `fullScreen`: Entire screen is the button
  3. `fakeLockScreen`: Black screen with minimal UI, touch-sensitive
     - Brightness reduced to near-zero
     - Wakelock active to prevent phone locking
     - Touch anywhere to start/maintain hold

### Configuration
- **Release sensitivity**: Per-mode configurable (not global)
  - Different modes can have different sensitivity thresholds
- **Haptic feedback on release**: Toggleable (default on)
  - User feels vibration when releasing button
- **Session start**: "Touch to begin" prompt (no auto-hold)
  - User must explicitly touch button to start engine
  - Prevents accidental session starts

### Behavior During Hold
- While held: grace period timer does NOT advance
- On release: grace period starts
- Must re-press before grace expires to continue
- Visual countdown during grace period (configurable to hide in stealth)

## Fake Call

### Call Styles
- **5 styles**:
  1. Android (native Android ringtone)
  2. iOS (native iOS ringtone)
  3. WhatsApp (WhatsApp default ringtone)
  4. Telegram (Telegram default ringtone)
  5. Signal (Signal default ringtone)
- **Ringtone selection**: Automatically matches chosen call style, defaulting to native OS behavior

### Answer Flow
1. Ringtone plays, screen displays incoming call
2. User taps "Answer"
3. Ringtone stops
4. Screen shows "Calling..."
5. Voice recording plays through earpiece (configurable to speaker)
6. User taps "Hang Up"
7. **Then**: Disarm occurs (chain resets to step 0)

### Decline Options
- **3 decline buttons** (configurable which are visible):
  1. **Disarm** (if `canDisarm=true`): Confirms user is safe, resets chain to step 0
  2. **Ignore** (default visible): Does NOT trigger miss count, allows dismissing notification
  3. **Distress** (long-press required): User holds 5 seconds
     - Progress ring appears at 1600ms
     - Triggers at 5s (like emergency call escalation)
     - Allows user to escalate if "Decline" gesture accidental

### Fake Call Decline Behavior
- **Decline counts as a miss** toward retryCount
- **On answer**: Chain pauses (does not escalate further)
- **On hang-up**: Disarm fires (reset to step 0)
- After `retryCount` misses (declines), chain advances to next step

### Audio & Voice
- **Voice recording**: User can record in-app OR pick from existing files
  - Max 2 minutes
  - Reasonable file size limit
- **Output mode**: Configurable (`voiceOutputMode`)
  - `earpiece` (default, private): Voice plays through earpiece only
  - `speaker` (default, audible to bystanders): Voice plays through speaker
- **Built-in voices**: 3 ringtones (Android, iOS, WhatsApp default) + per-language voice recording

### Session Behavior (Pivot 2 — fakeCall is event, not pause)
- **No engine pause on fake call**: The engine timer keeps running while `FakeCallScreen` is on top. `FakeCallScreen` is a route push, not a pause-and-overlay.
- **Hang-up timing**: Hang-up fires `disarm()` (reset to step 0). Decline follows the `declineIsSafe` semantics.

### Photo (NOT Used)
- Photo is NOT used for fake call UI
- Photo is reserved for emergency SMS (identification purposes only)

## Disguised Reminders

### Built-In Templates
- **Default templates**: Pre-loaded, can be disabled but NOT deleted
- **Rationale**: Ensures user always has options; prevents accidental removal of core features
- **Re-addable**: User can re-enable via "From template" picker

### Reminder Selection
- **Avoid repetition**: System avoids sending same template twice in a row
- **Pool**: User selects from all enabled templates (built-in + custom)

### Notification Styling
- **OS-matching**: iOS notifications styled as iOS native, Android as Android native
- **Innocuous names**: e.g., "Reminders" channel, not "Safety Alerts"
- **Background behavior**:
  - Simple types: Tap notification = check-in (closes notification)
  - `tapWord` type: Tap launches full-screen overlay (allows interaction)
  - Swipe noted in notification text (e.g., "Swiped notification")

### User Customization
- **Duration visible**: Configurable (shows/hides grace period countdown in notification)
- **Dismiss confirmation**: Requires realistic gesture (e.g., swipe), not "tap anywhere"
  - Prevents accidental dismissal by errant touch

## SMS Contact

### Message Composition
- **Placeholder system**: Supports dynamic insertion via buttons in text editor
  - `{name}`: Recipient name or "the owner of this phone"
  - `{location}`: Last known GPS coordinates (if enabled)
  - `{time}`: Current time
  - `{description}`: Session/event description
- **No name set**: Default to "the owner of this phone"
- **Default template**: "Automated safety alert from Guardian Angela.\n{name} may need help.\n..." (fully editable/deletable)
- **`{photo}` placeholder rejected (G-017):** not supported; MMS is unreliable and embedding photo URLs requires hosting/privacy infrastructure outside v3 GA scope. See spec 11 REJ-2 and spec 02 §6.

### Photo & Recording attachment (out of scope)
- **Photo**: NOT sent via SMS/messaging — see spec 11 REJ-2 (rejected).
- **Recording**: stored locally only, NOT attached to messages — see spec 11 REJ-2 (rejected, by design).

### Retry & Reliability
- **SMS retry**: Indefinite via native Kotlin WorkManager (Android)
- **iOS SMS limitation**: Opens Messages app; user must press Send manually (documented)
- **Non-blocking on failure**: Configurable per-step toggle (advanced option)
  - Default: block (wait for SMS to be queued before next step)
  - Can configure: continue chain regardless of SMS success

### Messaging Channels (Warning System)
- **WhatsApp**: Check installed at session start, warn user if not found
  - User must press Send manually (no auto-send)
  - Same placeholder system as SMS
- **Telegram**: Check installed, warn if not found
  - User must press Send manually
  - Same placeholder system as SMS

### No Fallback Chain Per Contact
- **Recommendation**: Add separate steps instead (e.g., WhatsApp step, then SMS step)
- **Rationale**: Simpler to understand and configure than implicit fallback

## Phone Call Contact

### Supported Call Channels
- **Phone call only** (tel: URI)
- **Rationale**: Deep links cannot initiate voice calls on WhatsApp/Telegram/Signal; phone call is the reliable channel

### Phone Call Behavior
- **Android**: Uses CALL_PHONE permission for auto-dial (no confirmation dialog)
- **iOS**: Always shows confirmation dialog (limitation, documented)
- **Server-side dispatch (Noonlight-style) is rejected** — see spec 11 REJ-4. Guardian Angela is offline-first by design; introducing a monitoring backend conflicts with the no-server invariant.

### Pre-SMS
- **Optional**: Send SMS before making voice call
- **Message system**: Same placeholder system as SMS contact
- **Retry delay**: Equals grace period (no separate configuration)

## Loud Alarm

### Disarm Policy
- **Always disarmable**: Loud alarm cannot be locked in. User must be able to stop it immediately.
- **Rationale**: If accidentally triggered, user must stop the alarm to prevent escalation
- **Gesture**: Tap button or swipe disarm slider

### Volume Control
- **Gradual volume increase**: Linear ramp over configurable duration (default 5 seconds per Q33)
- **System volume**: Set to max before playing (configurable toggle)
- **Override silent mode**: Alarm is the ONLY exception to silent/vibrate mode (configurable toggle)
  - Rationale: Silent mode must not prevent safety escalation

### Sound Options
- **Built-in sounds**: Siren only (Q9 keeps `siren` + `custom`).
- **Custom audio**: From files or in-app recording (`soundChoice: custom`)
- **Audio behavior**: Ducks (lowers) all other audio while alarm plays

### Photosensitivity
- **Screen flash**: Optional slower/safer variant for photosensitive users
- **Configurable**: Toggle in settings

## Call Emergency

### Confirmation Countdown
- **Default**: ON
- **Behavior**: 5-second countdown before dialing (gives last-chance cancel)
- **Rationale**: Safety feature prevents accidental emergency calls; countdown allows abort if triggered in error
- **Disarm available**: "I'm Safe" button available during countdown

### Non-Terminal Nature
- **NOT terminal**: Can have steps after emergency call in the chain
- **Behavior**: Call completes, chain continues to next step if configured

### Platform-Specific Behavior
- **Android**: CALL_PHONE permission for auto-dial
  - No confirmation dialog shown
- **iOS**: Always shows native confirmation dialog (limitation)
  - Cannot bypass; documented to user

### Emergency Number Management
- **Mapping**: 80+ country codes with default emergency numbers (112 as global fallback)
- **Editable**: User can customize per-mode
- **Validation warnings**: If user enters non-standard number, show warning but allow override
- **Recommendation**: Suggest locale-appropriate number

### Server-side dispatch — rejected
Noonlight-style monitoring backend integration is rejected for v3 GA; see spec 11 REJ-4. The offline-first invariant rules out a server dependency for core escalation, and a monitoring-centre tier is not on the v3 product roadmap.

## Hardware Button (Panic Trigger)

### Android Implementation
- **Method**: `dispatchKeyEvent` approach
  - Returns `true` without calling super
  - Suppresses volume change AND HUD (notification bubble)
- **Rejected alternative**: ContentObserver approach
  - Cannot detect key-down/up distinction
  - No long-press detection capability
  - Inferior to dispatchKeyEvent

### iOS Implementation
- **Method**: Audio framework + headphone remote detection (via audio_service)
- **Limitation**: Works only with connected headphones/remote (volume buttons cannot be detected)

### Trigger Patterns
- **One pattern at a time**: User configures either repeatPress OR longPress, not both
- **repeatPress**: Minimum 5 presses (up to 10)
- **longPress**: Duration from 1–10 seconds
- **Behavior**: Pressing configured pattern advances chain (manual escalation, not disarm)

### Testing & Preview
- **Test/preview available in settings**: User can verify detection works before relying on it

### Black Screen Behavior
- **When active**: Hardware button mode can trigger black screen display (optional)
- **UX**: Mimics locked phone (black screen + touch to start)

### Permissions
- Only requested when hardware button is configured in an active mode

## Stealth Mode

### StealthConfig Model
Stealth is configured via `StealthConfig` (not scattered boolean fields):

| Field | Type | Description |
|-------|------|-------------|
| `enabled` | bool | Master stealth switch |
| `fakeName` | String | Fake app name shown in notifications/header (default `'Music'`, Q20) |
| `fakeIcon` | `StealthIconPreset` | Enum: music, calendar, fitness, weather, news, photos, notes, clock, podcast, none (Q20) |
| `notificationDisguise` | bool | When true, the notification uses a generic channel name/icon (default `true`) |
| `timerDisplay` | enum | `normal` / `small` / `none` — how session timer is shown |
| `sessionScreenStealth` | bool | Whether session screen hides safety indicators |

Global: `AppDefaults.stealth`. Per-mode override: `ModeOverrides.stealth` (null = inherit).

### Stealth Features
1. **Fake music player UI**: Spotify-like appearance when opening app during stealth session
   - "I feel fine" disguised as track progress slider
   - No separate stealth toggle within the music player UI
2. **Notification disguise**: `notificationDisguise` configures text
   - Default: "Music playing"
   - Title, body, action button text all customizable
3. **Notification icon**: `fakeIcon` for neutral appearance (music player, etc.)
4. **Timer display**: `timerDisplay` enum: normal / small / none
5. **Session screen stealth**: `sessionScreenStealth` hides missed indicators, grace visuals,
   progress bar
6. **End screen**: Optionally suppressed (silent exit to home)

### Session End in Stealth
- Skip end screen (silent exit to home)
- Show "Session ended" summary (skippable)

## Simulation

### Core Decision: Practice Mode, Not Automated Playback

Simulation is a **practice mode** where the user interacts with the app exactly as they would in a real session. It is NOT a demo or automated preview that plays through the chain on its own.

The user MUST interact with every user-driven step:
- **holdButton**: User must hold the button. If released, the grace period runs and the chain escalates on timeout.
- **fakeCall**: User must answer or decline. If no interaction, the call times out and the chain escalates.
- **disguisedReminder**: User must dismiss the reminder. If no interaction, the miss is counted and retry/escalation logic runs.

There is NO auto-advance, NO auto-hold, NO auto-answer, and NO auto-dismiss in simulation.

### Speed Control
- **Range**: Up to 1000x via speed bar
- **Speed options**: 1x, 2x, 5x, 10x, 20x, 50x, 100x (+ custom slider)
- **Leap button**: Skip to 1 second before the next scheduled engine event (bypasses long wait phases without skipping interaction requirements)

### Silent Toggle
- **Per-session toggle** in the simulation controls bar; NOT persisted to storage.
- **Default**: OFF (audio plays normally for local-only steps)
- **When ON**: All audio suppressed — ringtone, voice recording, alarm, countdown sounds. Vibration still fires.
- **Loud alarm**: Always muted in simulation regardless of silent toggle.

### Scope of Simulation

**Fires normally — Silent=OFF (local-only):**
- Foreground service notification (title: "SIMULATION — [mode]")
- Disguised reminder notifications (with `[SIM]` suffix)
- Fake call screen + ringtone + voice recording
- Countdown warning UI + vibration + optional audio
- Location tracking

**Fires with Silent=ON:**
- Same as above, except all audio suppressed. Vibration and all UI still fire.

**BLOCKED in simulation (both silent modes, logged as `sim_blocked`):**
- SMS / WhatsApp / Telegram messages
- Phone calls to contacts
- Emergency calls
- Loud alarm audio (always muted; `[SIM]` card shown)
- Audio recording (privacy)

**Defense-in-depth**: 4 layers prevent real external actions in simulation:
1. `isSimulation` is `final` (immutable) on the engine.
2. Every `executeReal()` in each `EventStrategy` checks `isSimulation` first.
3. Every external-action service method accepts `isSimulation` and no-ops when true.
4. `SimulationMessagingService` / `SimulationPhoneService` subclasses structurally
   cannot reach real SMS/call code.

- **App kill watchdog**: NOT triggered in simulation

### Session Summary
- **Provided**: Includes timing breakdown, step progression
- **Shareable**: Can be shared/exported
- **Emphasis**: "SIMULATED" clearly marked
- **No "Start Real Session" button**: User must return home and start a real session intentionally.

### Loading Screen
- **Duration**: 1.5 seconds
- **Display**: Progress bar + "PRACTICE MODE" / "SIMULATED" emphasis
- **Purpose**: Prepare engine, show clear intent to user

### Validation
- **Lenient for simulation**: Warns but allows starting with missing contacts or permissions.
- **Strict for real sessions**: Blocks start only if the chain includes SMS/phone/emergency steps AND zero contacts are configured.
- **Preview in event defaults**: Same as simulation (can preview step before adding to chain)

- **Matching call style**: Ringtone reflects chosen call style (Android, iOS, WhatsApp, Telegram, Signal)
- **Built-in sounds**:

- **Ringer mode respected**: Silent/vibrate/volume settings honored
- **Exception**: Alarm is the only feature that overrides silent mode

- **Output**: Earpiece (default, private) OR speaker (audible)
- **Configurable**: Per-mode via `voiceOutputMode`
- **Built-in**: Pre-recorded voice per language

### Audio Encryption
- **At rest**: All audio files encrypted with innocuous filenames
- **Decryption**: On playback only
- **Rationale**: Privacy, plausible deniability in stealth scenarios

## Vibration

### Patterns
- **5 vibration patterns**:
  1. Warning: Pulse pattern indicating caution
  2. Confirm: Short double pulse (success feedback)
  3. Alarm: Rapid/intense pattern
  4. fakeCall (NEW): Pattern matching incoming call
  5. reminder (NEW): Pattern for reminder receipt
- **Configurable**: Per-event type OR globally
- **Silent mode respected**: Vibration respects phone silent mode
- **Exception**: Alarm vibration overrides silent mode (if enabled)

### No-Op Case
- **No vibrator hardware**: On devices without vibrator, vibration patterns are no-ops (silent failure)

## Notifications

### Channels (Android)
**4 separate notification channels** with distinct importance levels and behavior:

1. **Session Service** (Low priority)
   - Foreground service notification
   - Persistent, low visual prominence
   - Display name: "System Service" (innocuous)

2. **Reminders** (High priority)
   - Disguised reminders, check-in prompts
   - Display name: "Reminders" (innocuous, not "Safety Alerts")
   - Can wake screen if full-screen template

3. **Alarm** (Max priority)
   - Loud alarm escalation
   - Display name: "Alarm" (innocuous)
   - Can override DND (optional, configurable)

4. **Updates** (Default priority)
   - App updates, miscellaneous info
   - Display name: "Updates" (innocuous)

### iOS Notifications
- **Critical Alert entitlement**: Apply for safety app classification
- **Rationale**: Allows notifications to bypass DND for critical escalations
- **Requires**: App Store safety app category

### DND Behavior
- **Android**: Configurable toggle (default on) to override Do Not Disturb
  - Affects high-priority channels (reminders, alarm)

## Error Handling & Edge Cases

### SMS Failures
- **Indefinite retry queue**: Native Kotlin WorkManager manages queue persistently
- **No size limit**: WorkManager handles large queues
- **Non-blocking option**: Per-step toggle (can continue chain while SMS retried in background)

### Missing Contacts/Permissions
- **WhatsApp/Telegram/Signal check**: Verified at session start
- **Missing app**: Warn user, allow to disable channel for that contact
- **Continue anyway**: If user chooses to proceed, log failure if step is reached

### Permission Revoked Mid-Session
- **Continue chain**: Chain continues (does not crash)
- **Warn user**: Toast/notification of permission loss
- **Log failure**: Recorded in session log with reason
- **Step behavior**: Depends on `nonBlockingOnFailure` toggle

### No Signal / No SMS Service
- **Queue**: SMS added to WorkManager queue
- **Retry indefinitely**: WorkManager retries when signal returns
- **Continue chain**: If non-blocking enabled, proceed without waiting

### Low Battery
- **No power-saving mode**: Chain continues at normal speed

### Real Incoming Call
- **Timer pause**: Session timer pauses during real incoming call
- **Resumes**: When real call ends, session timer resumes
- **Fake call handling**: If real call arrives during fake call step, behavior undefined (log this edge case)

### App Kill During Session
- **Android watchdog notification**: AlarmManager watchdog triggers notification if app is killed
- **Restart needed**: User must explicitly restart (no auto-resume)
- **No persistent checkpoint**: Session state is in-memory only; killed session cannot be resumed

## Security Audit Recommendations

### Always-Encrypt (Mandatory)
- **Changed**: From optional to mandatory encryption of all persisted data (Drift database file + JSON-backed singletons)
- **Cipher**: `sqlite3mc` (AES-256) + AES-256 JSON envelope, sharing one key from `flutter_secure_storage`
- **Reason**: User privacy, plausible deniability in hostile situations

### Three Distinct PINs
- **App PIN** (`appPinHash`): Locks app at launch (before home screen). No biometric.
- **Session End PIN** (`sessionEndPinHash`): Required to disarm or end an active session.
  Timeout: 15s (configurable via `pinTimeoutSeconds`). Biometric MAY substitute.
- **Duress PIN** (`duressPinHash`): When entered at ANY PIN prompt, silently fires the
  selected distress chain without revealing to the attacker that it is the duress code.
- **All nullable**: null = disabled. Any combination is valid (e.g., only Session End PIN).

### App Access Control
- **App PIN**: Locks app at launch. No biometric (prevents forced fingerprint unlock on entry).
- **Session End PIN**: Required to disarm/end session. Biometric allowed as substitute.
- **Rationale**: Prevents unauthorized person from ending active safety session.

### Duress PIN
- **Duress PIN**: Third PIN hash in AppSettings. When entered at any PIN prompt, fires the mode's resolved distress mode silently (no error message shown to attacker). Engine is invoked via `replaceWithDistressChain(steps, triggerReason: TriggerReason.duressPin)`.
- **Rationale**: Safety if coerced to "unlock the app" or "stop the session."

### Wrong PIN Escalation
- **Attempts**: 5 wrong Session End PIN entries triggers the mode's resolved distress mode (`triggerReason: TriggerReason.wrongPinExhausted`).
- **Count**: Configurable threshold (default 5).
- **Reset**: Resets to 0 after successful entry or time-based reset.
- **Duress PIN does not count as wrong**: Silently escalates, no increment.

### Audio Recording Security
- **Encrypt at rest**: All recordings encrypted with innocuous filenames
- **No metadata**: Filenames don't indicate content
- **Decryption**: On playback only, in-memory
- **Deletion**: User can delete recordings at any time

### Data Deletion — rejected
Panic-wipe of session data is rejected (see spec 11 REJ-5). The user can manually uninstall the app to remove local data; emphasising backup/recovery serves the legitimate data-loss case better than a destructive one-tap.

### App Identity Concealment beyond stealth mode — rejected
Renaming the home-screen app label dynamically is rejected (see spec 11 REJ-6). The shipped stealth mode (`StealthConfig.fakeName` + `StealthConfig.fakeIcon` + activity-alias toggles on Android) covers the same threat model with less platform-API risk; full dynamic label-renaming is platform-locked and high-risk.

### Notification Channels
- **Innocuous names from start**: "System Service", "Reminders", "Alarm", "Updates" (never "Safety Alerts")
- **Configurable disguise**: Additional custom notification text

## Recent Decisions (Spec Conflicts Resolved)

### Real Phone Call Detection
- **Android**: `PhoneStateListener` detects incoming calls and auto-pauses session
- **iOS**: CXCallObserver detects calls (when audio is active) and auto-pauses session
- **Behavior**: Session automatically resumes when call ends, from exact point of interruption
- **UX**: No user action required; handled transparently

### Timing Configuration Grouping
- **All step types** support collapsible "Timing" section in mode editor
- Contains: `waitSeconds`, `durationSeconds`, `gracePeriodSeconds`, `retryCount`
- **Per-field consistency**: Every step has the same timing fields (some may not use all)
- **UX**: Reduces clutter in step editor; timing config grouped together

### Duplicate Step Button
- **Mode editor feature**: Each step can be duplicated with single click
- **Placement**: On each step card in the chain editor
- **Behavior**: Creates exact copy of step with all current config, inserted right after original
- **Use case**: Users can quickly create similar steps without manual re-entry

### Fake Call Answer Behavior (Pivot 2 — clarified)
- **On answer**: Engine keeps running — answer is a no-op at the engine level. The voice clip plays while escalation timers continue.
- **On hang-up**: Disarm fires (reset to step 0).
- **Decline**: Behaviour depends on `declineIsSafe` (default `true` ⇒ decline = disarm; `false` ⇒ decline = miss → grace → re-ring).
- **Rationale**: Pausing on every fake call would create gaps that an attacker could exploit; decline-with-distress lets the user escalate quickly if they hit decline accidentally instead of answering.
